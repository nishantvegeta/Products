## Technical Description

Enhances the order creation endpoint to support selecting billing and shipping addresses from the customer's saved addresses. Must implement the business rules:
- If explicit `billingAddressId` and/or `shippingAddressId` is provided, validate that the address belongs to the customer.
- If not provided, use the customer's default billing/shipping address (search for an address with `IsDefaultBilling=true` or `IsDefaultShipping=true` respectively).
- If neither explicit nor default address is available, return validation error.
- Address data must be copied into the order (immutable snapshot) at creation time. The order should store the address details directly (not just IDs) per the spec response shape which includes full AddressDto objects for billing and shipping.

Note: The current `Order` entity only stores `CustomerId`, `Product`, `Quantity`, `Price`, `Status`. It does NOT yet have fields for billing/shipping address details. This task requires extending the Order entity to add address-related properties.

## Approach

**Step A: Extend Order entity**
- Open `src/Products.Domain/Entities/Orders/Order.cs`
- Add properties for billing and shipping address details (since addresses are copied at order time, they become part of the order):
  - `BillingAddress` (string, JSON or concatenated? Actually spec shows structured object; but Order entity currently doesn't have these)
  - Better: Add detailed fields: `BillingAddressName`, `BillingAddressLine1`, `BillingCity`, etc., OR store as JSON string `BillingAddressJson`. Simpler: add the most commonly needed fields separately.
- However, looking at spec response, the order returns an `AddressDto` for billing and shipping. So we need to store enough info to reconstruct that. Options:
  1. Add separate columns for all address fields (billingName, billingAddressLine1, billingCity, ...). This is normalized and searchable.
  2. Store as JSON in a single column `BillingAddressDetails` (string) and deserialize. Simpler but less queryable.
Given that this is a small project and order history should be immutable, storing JSON might be acceptable. But the existing Order entity uses simple scalar fields only. To keep consistent with current pattern (Product stored as string), we could add fields:
   - `BillingAddress` (string? Actually could be just `BillingAddress` as `string` containing concatenated lines, but spec expects structured fields).
   - Wait: the spec's Order response includes:
     ```json
     "billingAddress": {
       "id": "...",
       "name": "...",
       "addressLine1": "...",
       "city": "...",
       "state": "...",
       "country": "...",
       "postalCode": "...",
       "phone": "..."
     }
     ```
     It includes `id`. That means we should store the address ID as well (to know which address was used). But note: the address may later be deleted/changed; the order keeps a snapshot. So we should store both the address ID (for reference) and the actual data fields (copied at time of order). The spec includes the ID in the response, meaning the order should retain the original address ID. So add `BillingAddressId` (Guid?) to Order.
   - Also store the copied data: maybe `BillingAddressName`, `BillingAddressLine1`, etc., or use a value object pattern.
   - For simplicity, add these properties to Order:
     - `BillingAddressId` (Guid?, nullable if not provided)
     - `BillingAddressName` (string, max 100)
     - `BillingAddressLine1` (string, max 200)
     - `BillingAddressLine2` (string, max 200, nullable)
     - `BillingCity` (string, max 100)
     - `BillingState` (string, max 100)
     - `BillingCountry` (string, max 100)
     - `BillingPostalCode` (string, max 20)
     - `BillingPhone` (string, max 20)
   - And same set for shipping: `ShippingAddressId`, `ShippingAddressName`, etc.
   - Also add `TotalAmount` (decimal) — computed as `Quantity * Price`, but might be stored.
- Update the Order entity class accordingly. Ensure it still inherits `FullAuditedAggregateRoot<Guid>`.
- Add these new fields to the DbContext configuration in `ProductsDbContext.OnModelCreating` with appropriate max lengths, same as other string fields.

**Step B: Extend OrderDto**
- Open `src/Products.Application.Contracts/Orders/OrderDto.cs`
- Add billing and shipping properties. They should be of type `AddressDto` (the same DTO we are creating for addresses). However, at the time of writing OrderDto may not have a reference to AddressDto because the Address DTOs haven't been created yet. After SHARED-004, `AddressDto` exists. So we can add:
  ```csharp
  public AddressDto BillingAddress { get; set; }
  public AddressDto ShippingAddress { get; set; }
  ```
  In the OrderDto class, ensure it includes these new properties. Update AutoMapper profile later to map from entity to these nested DTOs, but that's a separate task. For now, just ensure the DTO structure matches the spec response.

**Step C: Update CreateOrderDto**
- Open `src/Products.Application.Contracts/Orders/CreateOrderDto.cs`
- It already likely has `CustomerId`, `Product`, `Quantity`, `Price`. Add:
  - `Guid? BillingAddressId` (nullable)
  - `Guid? ShippingAddressId` (nullable)
- Ensure validation attributes match spec: `[Required]` for CustomerId, Product, Quantity, Price; optional for address IDs.

**Step D: Update OrderAppService**
- Open `src/Products.Application/AppServices/Orders/OrderAppService.cs`
- The `CreateAsync(CreateOrderDto input)` method exists. Enhance it to:
  1. Validate product, quantity, price as it already does.
  2. Validate `input.CustomerId` matches CurrentUser.Id.
  3. Resolve billing address:
     - If `input.BillingAddressId.HasValue`, fetch that address via `_addressRepository.FindAsync(input.BillingAddressId.Value)`.
       - If not found, throw 400 "Billing address not found"
       - If address.CustomerId != CurrentUser.Id, throw 400 "Billing address does not belong to customer"
       - Copy address data into a DTO or directly into order entity's billing fields
     - Else (not provided), find default billing address: `await _addressRepository.FirstOrDefaultAsync(a => a.CustomerId == CurrentUser.Id && a.IsDefaultBilling && a.IsActive)`.
       - If none, throw 400 "No billing address selected and no default set"
       - Copy data
  4. Resolve shipping address: same logic with `IsDefaultShipping`
  5. Set order's billing/shipping fields (either IDs or full copied fields as per entity design)
  6. Generate `OrderNumber` (e.g., `ORD-{DateTime.Now:yyyyMMdd}-{sequential}`; but existing Order entity may have auto-generation or not. Check if `Order` has `OrderNumber` property. The spec response includes `orderNumber`. If Order entity doesn't have one, add property `OrderNumber` and set it here. Need to generate unique number — can use a simple counter or timestamp + random. Since it's just a demo, maybe simple pattern.
  7. Save order: `await _orderRepository.InsertAsync(order);`
  8. Map order entity to `OrderDto` (which now includes BillingAddress and ShippingAddress as AddressDto). Ensure AutoMapper profile maps nested address entities to DTOs. If we stored address details directly, just map; if we stored address IDs, we may need to fetch addresses again to map to full DTOs, but ideally the order entity already contains the copied data fields, so we can map directly.
  9. Return success `ResponseDataDto<object>` with created OrderDto.

**Step E: AutoMapper Configuration**
- Update `src/Products.Application/ProductsApplicationAutoMapperProfile.cs` (or equivalent) to include mappings:
  - `CreateMap<CreateOrderDto, Order>()` — already exists? Need to extend to ignore new fields or map.
  - `CreateMap<Order, OrderDto>()` — include mapping for BillingAddress and ShippingAddress. If Order has BillingAddressName etc. fields, map them to nested `AddressDto` manually using `ForMember` and constructing new AddressDto from those fields. Or create a custom resolver.
  - `CreateMap<Address, AddressDto>()` likely already exists? Probably not, need to create. Actually there may be no existing mapping for addresses. Need to create `CreateMap<Address, AddressDto>()` and `CreateMap<CreateAddressDto, Address>()`, `CreateMap<UpdateAddressDto, Address>()`.
- If the auto mapper profile is in `Products.Application` project, add these mappings.

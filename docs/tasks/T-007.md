## Technical Description

Unit tests must verify that every address and order endpoint behaves as specified in the API spec — correct responses, validation, error cases, permission enforcement, and business rules. Tests should cover both success and failure scenarios.

---

## Approach

Create or extend `AddressAppServiceTests` in `test/Products.Application.Tests/Addresses/`. Use the existing test base from `Products.TestBase` for common setup: mock repositories, current user, mapper, logger. Use xUnit (or NUnit) and Moq for mocking.

Test class structure:

```csharp
public class AddressAppServiceTests : ProductsApplicationTestBase
{
    private readonly IAddressAppService _addressAppService;
    private readonly IRepository<Address, Guid> _addressRepository;
    private readonly IRepository<Customer, Guid> _customerRepository;
    private readonly IMapper _mapper;
    private readonly ICurrentUser _currentUser;

    // Setup: create test data (customer in DB), seed via DataSeeder
}
```

Implement test methods covering each endpoint scenario.

**Test Data Setup:**
- Seed a test customer in the database in `SetUp` or using fixtures.
- Set `_currentUser.Id` to the customer's Id for most tests.
- Use real repository with in-memory database (SQLite) as per test project pattern.

**Scenarios by endpoint:**

### CreateAsync
- `CreateAsync_WithValidInput_ShouldReturn200AndCreatedAddress()`: All required fields, should set IsActive=true, assign Id, CustomerId matches current user
- `CreateAsync_WithMissingRequiredField_ShouldThrowUserFriendlyException()`: Validate message
- `CreateAsync_WithDuplicateName_ShouldThrowUserFriendlyException()`: create one address with name "Home", then try another with same name
- `CreateAsync_WithAddressNameTooLong_ShouldThrowValidationException()`: name > 100 characters

### GetAsync
- `GetAsync_ExistingId_ShouldReturn200WithAddress()`: address belongs to current user
- `GetAsync_NonExistentId_ShouldReturn404()`
- `GetAsync_IdOfOtherCustomer_ShouldReturn404()`: security test

### GetListAsync
- `GetListAsync_Default_ShouldReturnOnlyActive()`: create both active and inactive, default filter returns only active
- `GetListAsync_IsActiveFalse_ShouldReturnOnlyInactive()`
- `GetListAsync_IsActiveNull_ShouldReturnAll()`
- `GetListAsync_Filter_ShouldSearchMultipleFields()`: filter by name "home" matches name, city? actually spec says name or address line. Test matching name, addressLine1
- `GetListAsync_Pagination_ShouldReturnCorrectPage()`: create 15 items, skip=5, max=5 returns 5 items, totalCount=15
- `GetListAsync_Sorting_ShouldOrderByName()`: create with different names, verify order

### UpdateAsync
- `UpdateAsync_ValidUpdate_ShouldReturn200AndUpdated()`: update name, city
- `UpdateAsync_NonExistentId_ShouldReturn404()`
- `UpdateAsync_IdOfOtherCustomer_ShouldReturn404()`
- `UpdateAsync_ChangeToDuplicateName_ShouldThrowUserFriendlyException()`: have two addresses, try rename second to first's name

### DeleteAsync
- `DeleteAsync_ValidId_ShouldSetIsActiveFalse()`: after delete, address.IsActive=false
- `DeleteAsync_NonExistentId_ShouldReturn404()`
- `DeleteAsync_IdOfOtherCustomer_ShouldReturn404()`
- `DeleteAsync_AddressUsedInActiveOrder_ShouldReturn400()`: create pending order referencing address, try delete -> fails

### CreateOrderAsync (OrderAppService)
- `CreateOrderAsync_WithExplicitAddresses_ShouldCreateOrderWithCopiedAddressData()`: provide billingAddressId and shippingAddressId, verify order's billing/shipping fields populated from address snapshot
- `CreateOrderAsync_WithDefaultAddresses_ShouldUseDefaults()`: don't provide IDs, but set defaults on address; verify used
- `CreateOrderAsync_MissingBothAndNoDefaults_ShouldReturn400()`: clear message
- `CreateOrderAsync_BillingAddressNotOwned_ShouldReturn400()`
- `CreateOrderAsync_ShippingAddressNotOwned_ShouldReturn400()`
- `CreateOrderAsync_AddressNotFound_ShouldReturn400()`
- `CreateOrderAsync_Valid_ShouldReturnOrderWithAddressDtos()`: verify response includes full AddressDto

**Mocking vs Real Repository:** Use real repository with in-memory DB to test EF Core behavior and constraints (like unique index). The existing tests likely use this approach. Follow that.

**Assertions:** Check ResponseDataDto<object>.Success, Code, Message, Data. For order, check TotalAmount = quantity*price, status = "Pending", order number format.

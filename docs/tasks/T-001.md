## Technical Description

Implements the address creation endpoint. Must enforce all validation rules from the spec: required fields, string length constraints, customer ID matching authenticated user, and address name uniqueness per customer. Must handle the business rules for default billing/shipping addresses: when a new address is marked as default, all other addresses of that type for the customer must be set to false.

---

## Approach

Create `AddressAppService` in `src/Products.Application/Addresses/` that implements `IAddressAppService`. Inject dependencies via constructor: `IRepository<Address, Guid>`, `ILogger<AddressAppService>`, `IMapper`, and `ICurrentUser` (to get authenticated user ID) or `IAbpSession`.

Implement `CreateAsync(CreateAddressDto input)` following the pattern from `CustomerAppService`:

1. Log method entry with input data
2. Call `ValidateInputAsync(input)` for validation (create this helper method)
   - Check all required fields (Name, AddressLine1, City, State, Country, PostalCode, Phone)
   - Verify CustomerId exists and matches the authenticated user's ID (throw `UserFriendlyException` with appropriate message if mismatch)
   - Check that the customer actually exists in the database
3. Check if an address with the same name already exists for this customer (query repository, exclude soft-deleted if needed) — throw `UserFriendlyException("Address name already exists for this customer")` if duplicate
4. Map DTO to entity using AutoMapper: `var address = _mapper.Map<CreateAddressDto, Address>(input);`
5. Handle default address business rules: if `input.IsDefaultBilling` is true, query all addresses for this customer where `IsDefaultBilling == true` and set them to false; same for `IsDefaultShipping`
6. Insert the new address via `await _addressRepository.InsertAsync(address);`
7. Map the saved entity (with generated Id) to `AddressDto` using AutoMapper
8. Return `new ResponseDataDto<object> { Success = true, Message = "Address created successfully", Data = dto, Code = 200 }`
9. Wrap in try-catch blocks: re-throw `UserFriendlyException` for validation errors, catch other exceptions and throw a generic friendly error with logging

Apply `[Authorize(AddressPermissions.Create)]` attribute on the method.

Use explicit unit of work if required by project conventions (some ABP apps use `[UnitOfWork]` attribute or explicit `IUnitOfWorkManager`). Check `CustomerAppService` — it doesn't use explicit UoW, so follow that pattern (ABP automatically wraps ApplicationService methods in UoW).

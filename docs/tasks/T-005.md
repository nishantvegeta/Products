## Technical Description

Implements the address deletion endpoint. According to the spec, this is a **soft delete** — sets `IsActive = false` rather than permanently removing the record. Must also enforce business rule: cannot delete if address is currently used in an active order. Must check ownership.

---

## Approach

Add `DeleteAsync(Guid id)` to `AddressAppService`. Follow pattern from CustomerAppService.DeleteAsync but with soft delete and reference check:

1. Fetch address by id
   - If null, return 404 ResponseDataDto
2. Check ownership: `if (address.CustomerId != CurrentUser.Id)` return 404 or 403
3. **Check if address is referenced by any active orders**:
   - Query `IOrderRepository` (or use `_orderRepository` if we inject it)
   - Find orders where `(BillingAddressId == id || ShippingAddressId == id) && order.Status == "Pending"` (or other active statuses as defined). Need to check what statuses count as "active". Spec says "active order" — likely orders that are not Completed or Cancelled.
   - Option: inject `IRepository<Order, Guid>` and check orders with status in `["Pending", "Processing", "Shipped"]` or similar.
   - If any such orders exist, return 400 with message "Cannot delete address because it is used in an active order"
4. Set `address.IsActive = false;`
5. Save: `await _addressRepository.UpdateAsync(address);` (Do not call DeleteAsync, because we want soft delete)
6. Return 200 ResponseDataDto with `Data = true` as per spec response.

Apply `[Authorize(AddressPermissions.Delete)]`.

Note: According to spec error responses include 403 and 404 but also "Cannot delete if address is currently used in an active order" as 400. Implement accordingly.

## Title

[Products E-commerce Platform] Add user story and scenarios for order status updates and processing workflow management

---

## User Story

As a Manager,
I want to view and update order statuses following the defined workflow,
So that I can process customer orders efficiently and maintain accurate order state with full audit trail.

---

## Scenario 1: Successful flow

Given I am an authenticated Manager with order processing permission
When I update an existing Order with status transition from "Pending" to "Confirmed"
And I provide any required notes or processing details
Then the Order.Status is updated to "Confirmed"
And an OrderStatusHistory record is created with: OrderId, OldStatus = "Pending", NewStatus = "Confirmed", ChangedBy = ManagerUserId, ChangeDate = current timestamp
And the Order.LastModificationTime is updated
And the response returns the updated OrderDto with current status and status history list
And the response status is 200 OK

---

## Scenario 2: Validation / error flow

Given I am an authenticated Manager
When I attempt to update an Order status from "Delivered" to "Draft" (irreversible transition)
Then I receive a 400 Bad Request response with validation error: "Status transition from Delivered to Draft is not allowed"
And the Order remains unchanged
And the response indicates which transitions are valid from current status

---

## Scenario 3: Alternate flow

Given I am an authenticated Manager
When I query for orders to process using filters: Status = "Pending", CreatedDate within last 7 days, sorted by OrderDate ascending
Then I receive a paginated list of orders matching those criteria
Each order includes basic details: OrderId, OrderNumber, CustomerName, TotalAmount, Status, OrderDate
The list is sorted by OrderDate ascending (oldest first) for priority processing
And the pagination metadata shows total pending orders count

## Acceptance Criteria

- Manager can view all orders (across customers) with filtering and sorting
- Manager can update Order.Status following the defined workflow: Draft → Pending → Confirmed → Processing → Shipped → Delivered (some transitions irreversible as per BR-002)
- Each status change must create an audit record in OrderStatusHistory (OrderId, OldStatus, NewStatus, ChangedByUserId, ChangeDate, Notes)
- Managers cannot modify orders belonging to other managers unless business rule allows (role-based, typically all orders visible)
- Order update endpoint validates transition rules and returns 400 with message for invalid transitions
- Order queries exclude soft-deleted orders (IsDeleted = false filter)
- The system logs all status change events with user ID and timestamp (audit trail)
- Order view list supports pagination (default 20, max 100), sorting (OrderDate, Status, TotalAmount), and filtering (Status, CustomerId, date range)
- Manager can add notes or comments to order updates (optional)
- Concurrency control: optimistic concurrency using RowVersion prevents conflicting updates
- All monetary fields (TotalAmount, etc.) remain unchanged after order creation (except calculated fields like tax adjustments if allowed)
- The endpoint uses ABP's AppService patterns with proper DTOs (UpdateOrderDto, OrderDto, OrderListDto)

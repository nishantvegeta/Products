## Title

[Products E-commerce Platform] Add user story and scenarios for order creation with stock validation and price locking

---

## User Story

As a Customer,
I want to create a new order with selected products and quantities,
So that I can purchase products while ensuring price accuracy and stock availability at order time.

---

## Scenario 1: Successful flow

Given I am an authenticated Customer with selected products in my cart
When I submit a CreateOrderDto with: CustomerId (own ID), BillingAddressId, ShippingAddressId, and OrderItems (ProductId, Quantity for each)
Then the system validates that all ProductIds exist and are active
And the system checks stock quantities: for each item, Quantity ≤ Product.StockQuantity
And the system locks prices by copying Product.Price to OrderItem.UnitPrice at this moment
And an Order is created with status = "Draft" (or "Pending" based on workflow)
And OrderItems are created with proper foreign key references to Product entity
And the TotalAmount is calculated as sum of (OrderItem.UnitPrice × OrderItem.Quantity) plus shipping/tax
And the response returns the created OrderDto with OrderId, TotalAmount, and OrderItem details
And stock quantities are not yet reduced (reserve on confirmation) or reduced based on business decision
And the response status is 201 Created

---

## Scenario 2: Validation / error flow

Given I am an authenticated Customer
When I submit a CreateOrderDto with OrderItems where one product has Quantity (10) exceeding AvailableStock (5)
Then I receive a 400 Bad Request response with validation error: "Insufficient stock for Product [SKU]"
And no Order is created
And stock quantities remain unchanged
And the response includes details about which product and requested vs available quantity

---

## Scenario 3: Alternate flow

Given I am an authenticated Customer placing an order
When I omit the ShippingAddressId and only provide BillingAddressId
Then the system uses BillingAddressId for both billing and shipping (same address)
And the Order is created successfully with ShippingAddressId = BillingAddressId
And the response returns OrderDto with both address fields populated with the same AddressId
And the response status is 201 Created

## Acceptance Criteria

- The CreateOrder API endpoint requires authentication
- Customer can only create orders for themselves (CustomerId must match authenticated user or be derived automatically)
- Each OrderItem must store: ProductId (FK to Product), ProductName (snapshot for historical display), UnitPrice (locked at order time, decimal), Quantity, LineTotal (calculated)
- The system enforces: Quantity > 0 for all items
- The system validates: all ProductIds exist, are active (IsDeleted = false), and Product.StockQuantity ≥ Quantity
- Price locking: OrderItem.UnitPrice must equal Product.Price at the moment of order creation (do not rely on FK alone)
- Order status starts at "Draft" and follows the defined workflow
- TotalAmount field is calculated and stored on the Order (sum of LineTotals + shipping + tax)
- The response includes full OrderDto with nested OrderItems, BillingAddress, ShippingAddress, and Customer information
- The system logs the order creation event with user ID and order ID (audit)
- Concurrency control: use RowVersion or optimistic concurrency to prevent race conditions on stock updates
- The endpoint uses ABP's CrudAppService or custom AppService with proper DTOs (CreateOrderDto, OrderDto)

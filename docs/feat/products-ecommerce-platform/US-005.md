## Title

[Products E-commerce Platform] Add user story and scenarios for viewing order history with status tracking

---

## User Story

As a Customer,
I want to view my full order history with current status and details,
So that I can track my purchases, see order status, and access order summaries.

---

## Scenario 1: Successful flow

Given I am an authenticated Customer with multiple existing orders
When I request my order history with pagination (page 1, size 10) and sorting by OrderDate descending
Then I receive a paginated list of my orders (excluding deleted/soft-deleted orders)
Each order entry includes: OrderId, OrderDate, TotalAmount, Status (e.g., Draft, Pending, Confirmed, Shipped, Delivered, Cancelled)
The list is ordered by OrderDate descending (newest first)
The pagination metadata shows total count and current page
And the response status is 200 OK

---

## Scenario 2: Validation / error flow

Given I am an authenticated Customer
When I request my order history with an invalid page parameter (e.g., negative page number)
Then I receive a 400 Bad Request response with a validation error message
And no order data is returned
The error message indicates the valid parameter constraints

---

## Scenario 3: Alternate flow (filtering)

Given I am an authenticated Customer with orders in different statuses
When I filter my order history by status = "Shipped"
Then I receive only orders where Status = "Shipped"
Orders with other statuses (Draft, Pending, Delivered, etc.) are excluded
The pagination and sorting still apply to the filtered result set
And the response includes the filtered count in pagination metadata

---

## Acceptance Criteria

- The API endpoint GET /api/app/customer/orders requires authentication
- Customer can only retrieve their own orders (filtered by CustomerId = authenticated user ID automatically)
- The endpoint supports pagination parameters: Skip/Page, MaxResultCount/PageSize (default 10, max 100)
- The endpoint supports sorting by OrderDate, TotalAmount, Status (ascending or descending)
- The endpoint supports optional filtering by OrderStatus (single value or multiple)
- Orders that are soft-deleted (IsDeleted = true) are excluded from results
- Each OrderDto includes at minimum: OrderId, OrderNumber (if applicable), OrderDate, TotalAmount, Status, BillingAddress summary, ShippingAddress summary
- Nested OrderItems may be included in a separate detail endpoint or optionally loaded via query parameter (e.g., includeItems=true)
- The response follows ABP's PagedResultDto format with result array and total count
- Business rule: only show orders that belong to the authenticated customer (BR-005 enforced)
- The system uses repository filters to ensure data isolation (no cross-customer access)
- Performance: queries use proper indexing on CustomerId and OrderDate for fast retrieval
